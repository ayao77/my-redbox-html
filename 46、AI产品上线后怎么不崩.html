上线后 30 天 · 质量分走势质量成本
你以为:上线就稳了,一条直线。
上线第一天就要盯的 4 类
质量
幻觉率、答得对不对。用 LLM 当裁判在线抽样打分 + 语义相似度。
性能
首字延迟 TTFT、p95、失败率。参考线:好的模型 TTFT < 600ms。
成本
每请求/每用户/每功能的 token 和钱。输出 token 比输入贵 3–5 倍。
安全
护栏拦截率、毒性、越界、PII 泄露。出事先于用户发现。
核心 · 3 个上线后的硬功夫
A

生产 Eval(别只上线前测)

线上真实流量也要打分
对线上真实流量抽样打分 + 维护一个回归集(攒翻车 case)。分数跌破阈值就告警。研究说提前抓回归能拿 30%+ 准确率提升。
B

Prompt 当代码管

版本化 · 能回滚 · 先过回归
改 prompt 不是直接上 —— 要版本化、能回滚,改完先过回归 Eval 再发。有个研究就叫"更好的 prompt 反而把效果改坏了",别凭感觉改。
C

链路追踪 Tracing

出问题能查到哪一步
多步 / Agent 用 correlation ID 把每一步串起来。出幻觉、上下文丢了、变慢了,顺着 trace 就能定位是检索、还是某次工具调用出的问题。
怎么用 · 走一遍
一次最常见的上线后翻车 —— 你什么都没做,它却崩了。
第 7 天:供应商把模型从 4o 悄悄升到新版,你没动任何代码。
1
质量分悄悄从 92 掉到 78(漂移),用户开始说"它变蠢了"。
2
drift 告警触发(质量分跌破基线 5 分)→ 推到 Slack / PagerDuty。
3
先止血:回滚到锁定的模型版本,质量分拉回 92。
4
把新版加进回归集重测,定位掉分原因,修好再灰度上。
PM

上线运维 · 拍板默认值

能直接抄去跟工程提
① 在线打分抽样率 —— 全量太贵,抽样够用。默认:5%–10% 真实流量在线打分
② 告警阈值 —— 别等用户投诉。默认:质量分跌破基线 5 分 / 成本超日预算 → 告警
③ 回归集 —— 持续养。默认:先攒 50–100 条真实 case,翻车样本随时加
④ 模型版本 —— 别随供应商飘。默认:锁定版本,不自动升级;升级先过回归
⌨ MVP 三步(周一就能开干)① 接 Langfuse(开源)埋 trace + token 成本 → ② 攒回归集 + 在线抽样让 LLM 当裁判打分 → ③ 配"质量/成本跌破阈值 → Slack 告警"。先把「看得见 + 会报警」跑起来。
