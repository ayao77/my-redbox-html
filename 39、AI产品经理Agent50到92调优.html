METHOD 1 · TOOL DESCRIPTION

1Tool Description · 工具说明书

Tool Description 翻译过来「工具说明书」 · 也就是 — 你给 AI 用的每个工具 · 得写一段说明书告诉它「这工具干啥 / 啥时候调 / 参数咋填」
9 成工程师默认随便写一行(比如就写「查询订单」)→ AI 看不懂啥时候该调 · 经常乱调
第 1 段
What for
何时调
「用户问订单状态时调」
第 2 段
When not
不调的边界
「用户闲聊不调」
第 3 段
Example
参数示例
「order_id 是 8 位数字」
★ 真实测试数据(BENCHMARK)
默认 1 行说明
50%
三段式说明书 ★
62%
+ 12 分 · 0 成本 · 只是改文字
PM 一句话指令
「每个 Tool Description 改三段式 · 何时调 / 不调 / 参数示例 · 跑一周看成功率」
METHOD 2 · REACT

2ReAct · 让 AI 边想边干

ReAct = Reasoning(推理) + Acting(行动) · 翻译过来「思考 + 行动」
也就是 — 不是 LLM(大语言模型 · ChatGPT 背后那种)直接调工具 · 而是 LLM 先写一段 Thought(思考过程):「我要干啥 / 需要哪个工具 / 为啥选它 / 参数填啥」· 然后再调
★ 真实测试数据
直接调用 · 不思考
62%
ReAct 边想边干 ★
77%
+ 15 分 · 5 个手法里单点最猛
真实案例 某 BI(商业智能 · 公司看数据的工具)Agent · 用户问「华东上周销售」 · 默认直接调 SQL(跟数据库说话的语言)工具 · 经常 SQL 写错 · 加 ReAct 让 AI 先思考「需要哪几张表 / 怎么 join」 → SQL 一次写对率 89%
💡 ReAct 这一节工程师要看 3 遍 · 先截图保存
METHOD 3 · PRE-CALL VALIDATION

3调用前校验 · 防上线事故

Pre-call Validation 翻译过来「调用前校验」 · 也就是 — 调工具前 · 用规则或小模型先检查参数合不合理
LLM 经常生成「看着对其实错」的参数:订单金额负数 / 转账目标账户为空 / 库存不够还下单
★ 真实测试数据
不校验直接调
77%
加调用前校验 ★
87%
+ 10 分 · 还防上线事故
真实案例 某电商 Agent 自动发优惠券 · 默认 LLM 生成的金额没限制 · 几次生成「1 万块的券」直接发出去赔几十万 · 加调用前校验「金额必须 < 商品价 × 30%」→ 事故归零
PM 一句话指令
「每个高风险工具加 Pre-call 规则层 · 列死硬约束(金额上限 / 必填字段 / 状态依赖)」
METHOD 4 · MEMORY 分层

4Memory · 短期 + 长期记忆

Memory 翻译过来「记忆」 · Agent 记忆分两层
· Short-term Memory 短期记忆 · 也就是「当前对话内的上下文」(LLM 自带 Context Window(上下文窗口 · LLM 一次能看多少字)自动管)
· Long-term Memory 长期记忆 · 也就是「跨对话保留的用户偏好 / 历史」
9 成 Agent 只做短期 → 用户开新对话就忘 → 每次重新自我介绍 · 体验差
短期
Short-term
单次对话内
LLM Context Window 自动管 · 不用操心
长期
Long-term
跨对话保留
存 Vector DB(向量数据库 · 专门存数字化文本)· 用户开新对话 Agent 自动检索
★ 真实测试数据
只有短期记忆
87%
加长期记忆 ★
89%
+ 2 分 · NPS 涨 15 点 · 体验质变
真实案例 用户上周跟 Agent 聊过「想买电动车」· 这周问「最近啥推荐」· 没长期记忆 Agent 只能泛推 · 有长期记忆 Agent 说「上周您聊过想买电动车 · 这几款您可以看看」· 体验质变 · NPS(Net Promoter Score · 用户推荐意愿)涨 15 点
💾 这一节保存 · PRD 里直接抄
METHOD 5 · SELF-REFLECTION

5Self-reflection · 让 AI 自检

Self-reflection 翻译过来「自我反思」 · 也就是 — 任务做完后 · 让 Agent 自己再检查一遍「这做对了吗」· 发现错就回滚 · 重试
做法:加一段 Prompt — 「请检查上一步是否满足用户需求 · 不满足说明哪里错」 · LLM 自检后自动重试 / 或标记「不确定」让人工接手
