第 06 讲｜AI 落地避坑

这 3 种场景
千万别上 RAG

By NOVA · 硬上 = 浪费成本 + 体验更差
01
SCENE · ONE
强时效场景
这种场景长啥样
数据每天甚至每秒在变

用户要的是"此时此刻最新的答案"。RAG 检索的是静态文档，根本追不上数据变化的速度。

真实例子
用户问 "今天苹果股价多少？"
用户问 "北京现在天气怎么样？"
用户问 "我最近一笔订单到哪了？"

这些问题每分钟答案都不一样。你提前灌进 RAG 的数据，从存进去那一刻就已经过期。

正确做法
用 Function Calling 直接调实时 API
→ 股价 → 调 金融行情 API，实时拉数据
→ 天气 → 调 和风天气 / 高德 API
→ 订单 → 调 公司内部订单系统 API
硬上 RAG 的代价
用户体验崩 + 维护成本爆炸
答案永远是旧数据，用户骂"你这 AI 是不是傻"
为了减少"过期"，要不停更新向量库，运维成本飙升
同样一个问题，明明一次 API 调用就解决，硬走 5 步流程
02
SCENE · TWO
强推理场景
这种场景长啥样
要的不是查资料，是思考

RAG 的本质是"找资料 + 复述"。但有些问题，资料就在那儿摆着，关键是要分析、对比、推导。

真实例子
用户问 "帮我分析这份财报的 3 个潜在风险"
用户问 "对比这 2 个方案，哪个 ROI 更高"
用户问 "根据用户画像，推荐合适的策略"

这些问题不需要外部资料。模型已经具备所有信息，需要的是推理能力，不是检索能力。

正确做法
用 推理型大模型 + CoT
→ 模型选 DeepSeek-R1 / GPT-o3（推理能力强）
→ Prompt 加 "一步一步思考"（CoT 思维链）
→ 必要时上 Multi-Agent，让多个 Agent 各自分析再汇总
硬上 RAG 的代价
答案浅 + 绕弯路
RAG 只会把财报某些段落原样搬出来，没有真正的"分析"
本来一次推理就能解决，硬走"切段-向量化-检索-拼 Prompt"4 步
反而限制了模型的推理空间——它被检索结果"框住"了
03
SCENE · THREE
短文档场景
这种场景长啥样
文档总量少于 10 万字

现在主流模型的上下文窗口都很大（Claude / Gemini 都支持 100 万 token）。文档少的话，直接全塞进 Prompt 就行。

真实例子
场景 公司退货政策一共 3000 字
场景 产品说明书 5 页
场景 员工手册 20 页

这种场景搭 RAG，属于过度设计。你在用大炮打蚊子。

正确做法
直接 塞 Prompt 或用长上下文模型
→ 系统 Prompt 里直接贴全文，每次提问都带上
→ 用 Claude / Gemini 长上下文模型，一次性放 50 万字也没问题
→ 配合 Prompt Caching（提示词缓存），token 成本降 90%
硬上 RAG 的代价
工程量翻 10 倍 + 效果反而更差
本来 1 天能上线，硬搭 RAG 要花 2 周建库 / 调优
检索环节引入额外误差——召回不准，反而漏信息
维护一套向量数据库的运维成本完全没必要

面试金句 · 直接抄
被问"这个场景该不该上 RAG"——
第 1 步问 数据时效性高不高
第 2 步问 是检索还是推理任务
第 3 步问 文档量有多大
三个都不踩雷，才上 RAG

能答出"分场景选型"四个字 → 面试官知道你真踩过坑

核心结论 · TAKEAWAY
RAG 不是银弹
是特定场景的工具
