026 种死法:多 Agent 在哪翻车
这期的核心资产。每种都配 场景 → 根因 → 兜底。
记忆孤岛 / 上下文不一致
场景用户跟导购说"40 码/预算 300/不要赠品",转客服全忘了重问;导购说"有货",履约一查没货。
根因每个 Agent 各存各的记忆,没有单一事实源。
兜底共享上下文层(single source of truth)——会话/用户档案/订单/库存,所有 Agent 读同一份。
边界冲突 / 踢皮球
场景退款请求,导购推客服、客服推售后,用户转一圈回原点;或客服改地址、营销又叠券,价格算错。
根因角色边界没定清,谁能做什么没白名单。
兜底主管编排 + 角色权限白名单(客服永不能改价)+ handoff 协议(交接带完整上下文)。
协作死锁 / 无限循环
场景导购等履约确认、履约又回头等导购澄清,互相等;或任务在两 Agent 间反复横跳不收敛。
根因没设最大跳数 / 步数上限,没循环检测。
兜底最大跳数 + 循环检测 + 超时熔断。
成本 / 延迟爆炸
场景一个"查订单 + 改地址",在四五个 Agent 间传十几次、每次都调大模型,用户等 3 分钟,月底账单爆。
根因能单 Agent 干的拆成了多 Agent,跳数 = 成本。
兜底能单 Agent 别上多 Agent + 预算/跳数硬墙 + 缓存高频链路 + 简单任务不进多 Agent。
错误级联放大
场景导购把"不要赠品"理解成"要赠品",下游履约照发、客服照确认、营销照追加——一个错放大成一串。
根因上游一错,下游全盘接受,没中途校验。
兜底关键节点加审阅 Agent(Reviewer)+ 关键动作二次确认 + 失败隔离,不让一个错扩散。
责任模糊 / 不可观测
场景订单出错,CTO 问"哪个 Agent 的锅",没人答得上,没日志没 trace,没法复盘。
根因没有全链路追踪,Agent 间输入输出没留痕。
兜底全链路 trace + 责任归属日志 + 可观测面板,定位到具体 Agent。
03怎么搭:编排三步 + 范式选型
先搭地基,再加 Agent。顺序错了全白搭。
1
先问要不要多 Agent
能单 Agent + 工具调用解决就别上。多一个 Agent = 调用翻倍 + 故障面翻倍。
2
先搭共享上下文层
单一事实源:会话 / 用户 / 订单 / 库存,所有 Agent 读写同一份。治记忆孤岛的根。
3
定编排 + 边界 + 兜底
选范式 + 角色权限白名单 + 最大跳数 + 熔断 + 审阅 Agent + 全链路 trace。
编排范式 4 选 1(按问题选,不要硬套)

orchestrator → 子Agent
一个主管决定调谁。适合客服分流。

导购→下单→履约→通知
固定顺序,适合标准下单流程。
Parallel 并行

多源同时查,适合聚合查询。
Hierarchical 分层
Planner→Worker→Reviewer
复杂任务,规划者分派执行+审阅。

PM 盯这 7 个"协作级"指标(不是单 Agent 准确率)
① 端到端完成率
整条链跑通,不是单 Agent 答对
② 转交成功率
handoff 没丢上下文的比例
③ 上下文一致性率
跨 Agent 同一事实是否一致(对应 F1,最关键)
④ 循环/死锁率·平均跳数
对应 F3/F4
⑤ 端到端延迟·单任务成本
对应 F4
⑥ 级联错误率
一个错引发的连锁,对应 F5
⑦ 可追溯率
出错能否定位到具体 Agent,对应 F6
被问"多智能体怎么落地",按 4 句答:
1先问要不要——能单 Agent 别上,多 Agent 调用翻倍、故障面翻倍。
2先搭共享上下文层——single source of truth,治记忆孤岛。
3先定编排+边界——Supervisor 主管路由 + 角色白名单 + handoff 带上下文。
4先设兜底——最大跳数 + 熔断 + 审阅 Agent + 全链路 trace。
4 句答完,面试官知道你不是被 demo 带节奏的小白。
多智能体不是越多越强——是越多越容易各说各话

多 Agent 翻车,不是 Agent 不够聪明,是它们没共用一个脑子。Agent 数量是成本,协作质量才是产出。先搭共享上下文层,再加 Agent。
