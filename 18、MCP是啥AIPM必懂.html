MCP 是啥
为啥AI PM 必须懂

By NOVA · 不讲协议规范 · 只讲 PM 视角
2025-2026 最重要协议
你以为 MCP 是工程师才需要懂的协议——
错。
MCP 直接改写你设计 Agent 产品的方式
不懂 MCP 的 PM——还在为每个工具写 Function Calling。懂 MCP 的 PM——已经在搭"AI 应用商店"了。这是认知代差。
01
PART · ONE
MCP 到底解决了什么

先不讲"MCP 是 Model Context Protocol"这种废话。直接看一个真实场景——你给 AI 接入 5 个工具，没 MCP 之前 vs 有 MCP 之后。

BEFORE 没 MCP
之前
每接入一个新工具 → PM 写 PRD + 工程师写 Function Calling 接口
5 个工具 = 5 套 FC schema + 5 套权限管理 + 5 套错误处理
工具方升级 API → 所有接入方都要改，谁先改谁先挂
你的 Agent 只能用你接过的工具，AI 应用之间无法共享能力
AFTER 有了
MCP
工具方自己出一个 MCP Server（Notion / GitHub / 飞书都已经做了）
你的 AI 应用一行配置直接接入，能力自动发现，不用写 FC
工具方升级 → 改自己的 Server，客户端零修改
新装一个第三方 MCP Server，AI 立刻有新能力——像装 App 一样装能力
MCP 之于 AI = USB-C 之于电子设备
USB-C 出来之前——苹果有 Lightning、安卓有 micro USB、笔记本有自家接口。每接一个设备要换一根线。USB-C 出来之后——一根线连所有。

MCP 就是 AI 世界的 USB-C。工具方做 1 个 MCP Server，所有 AI 应用直接用。
2025-2026 谁都在用——
不是 Anthropic 一家的事

已经接入 MCP 的：Claude Desktop、Cursor、Cline、Continue、ChatGPT、Zed、Windsurf、飞书、阿里通义、字节豆包 IDE。

提供 MCP Server 的工具方：GitHub、GitLab、Notion、Slack、Linear、Sentry、PostgreSQL、Stripe、Cloudflare、Anthropic 自家全套。

这不是某家公司的小协议——是 AI 时代的"USB-C 标准"。不懂就被时代抛下。

02
PART · TWO
MCP vs Function Calling 怎么选

不是 MCP 取代 Function Calling——是不同场景用不同的。PM 必须知道什么时候用哪个。

PM 决策点

一句话 PM 决策心法

对内 + 性能敏感 → Function Calling

对外 + 生态扩展 → MCP

大部分高级 PM 的真实选择——两者混用。核心闭环用 FC，第三方扩展用 MCP。

最容易踩的坑——所有工具都上 MCP

有的 PM 听说 MCP 火，把核心高频内部工具也改成 MCP——延迟翻 2-3 倍，错误率上升，被工程师骂死。

记住——MCP 是为"互不相识的工具方和 AI 方"设计的。你和你自家工具不需要"互不相识"。

03
PART · THREE
MCP 改变产品架构的 3 个场景

光看协议没用——看 MCP 怎么改变你做产品的方式。3 个真实场景，PM 能立刻看到收益。

CASE 01 · 内部 Agent
公司 Agent 接30 个内部系统
SCENARIO · 中大厂内部 AI 助手
没 MCP 之前
每接一个内部系统（HR、ERP、报销、知识库、监控...）—— PM 写 PRD → 工程写 FC → 测试 → 上线。30 个系统 = 30 个轮回。新增一个系统排期 2 周。
用 MCP 之后
建一个公司内部 MCP Marketplace——各业务团队自己出 MCP Server 入驻。AI 应用方0 改动接新工具。从 PM 推驱动变 业务方自助。
PM 视角的真收益
PM 不用排队接需求了——你成为"平台架构师"，定义协议和准入标准，放大杠杆几十倍。这是 PM 的角色升级。
CASE 02 · 客户支持
支持 Agent 跨 Zendesk + Slack + Jira 流转
SCENARIO · ToB SaaS 客户支持自动化
没 MCP 之前
客户在 Zendesk 提工单 → 你写 FC 把信息拉过来 → AI 判断 → 写 FC 同步到 Slack 群 → 升级写 FC 建 Jira。3 个工具 3 套接口，任何一家升级 API 都断链。
用 MCP 之后
Zendesk、Slack、Jira 都官方出 MCP Server。你的 Agent 配置文件里写三行，能力直接到位。三家任何升级，你完全无感。
PM 视角的真收益
维护成本砍 80%——以前每月要花 1 周修各种接口断链，现在 0。省下的时间用来设计更复杂的工作流。这就是 MCP 的杠杆。
CASE 03 · 用户自助
让用户自己给你的 AI 装能力
SCENARIO · AI 应用市场化策略
没 MCP 之前
用户要新功能——提需求 → PM 评估 → 工程排期 → 上线。用户能装的功能 = PM 排期决定。AI 应用同质化严重。
用 MCP 之后
用户在你的 App 里一键添加 MCP Server——想接 Linear？装 Linear MCP。想接自家 ERP？技术员自己写一个。能力从 PM 中心化变成生态化。
PM 视角的真收益
你做的不再是"一个应用"——是"一个能力承载平台"。用户黏性指数级上升（装了 N 个 MCP 后切换成本极高）。Cursor 就是这么起来的。
面试金句 · 直接抄
被问"MCP 和 Function Calling 有啥区别"——
协议层——MCP 是开放标准，FC 是模型私有接口
扩展性——MCP 装即用，FC 必须写代码
解耦度——MCP 工具方升级客户端零改，FC 双端同步
PM 选型——对内 + 性能敏感用 FC，对外 + 生态用 MCP

4 句答完，面试官知道你跟得上 2025-2026 节奏

核心结论 · TAKEAWAY
MCP 改变的不是 Agent 技术——
是 PM 的角色边界
