Function Calling
到底怎么设计

By NOVA · 让 AI 真正能"动手"的核心机制
01
PART · ONE
怎么定义工具
先讲清是什么
Function Calling = 让 AI 能调用外部工具

大模型本身只会说话。要让它查数据、调 API、下订单、发邮件——必须靠 Function Calling 这套机制。

这是 AI 从"能说"变"能做"的分水岭。

核心机制
本质上是3 件事
你提前告诉 AI："这些工具可用，每个工具能做啥"
AI 自己判断用户的话该调用哪个工具
AI 生成结构化参数，你拿去真的执行
完整调用流程 · 5 步
1
用户说："北京今天天气怎么样"
2
AI 判断：需要调 get_weather 工具
3
AI 生成参数：{"city": "北京"}
4
后端真的调 API → 拿到 "晴 22°C"
5
AI 把结果组织成自然语言回给用户
工具定义标准格式
用 JSON Schema 描述每个工具
{ "name": "get_weather", "description": "查询某城市的实时天气", "parameters": { "city": { "type": "string", "description": "中文城市名", "required": true } } }

三个字段必须有：name（工具名）、description（用途）、parameters（参数 Schema）。

02
PART · TWO
怎么让 AI 选对工具
最核心的认知
工具描述写得好不好，决定了 AI 调用的准确率

AI 不是看你工具的名字选，是看 description 字段的语义匹配。description 写得模糊 → AI 选错；写得清晰 → AI 选对。

这是 90% 的人不知道的细节。

差 模糊描述
"description":
"查询信息"
AI 不知道查什么信息、什么时候用。准确率掉到 50%。
优 精准描述
"description":
"查询某城市未来 7 天天气预报。
仅支持中国城市。
用户问'明天会下雨吗'
'这周天气如何'时用。"
AI 一眼看懂用什么场景。准确率 90%+。
3 条黄金法则
description 必须包含
工具做什么（动词开头，明确动作）
适用场景（什么时候用 / 用户什么话术触发）
边界条件（什么情况不能用 / 限制是什么）
3 个最常踩的坑
工具名起得太抽象——比如 handle_request。AI 不知道这是干啥的。
工具太多——一次给 AI 20 个工具，准确率断崖下跌。建议 ≤ 10 个。
工具职责重叠——两个工具能做同一件事，AI 会随机选。
老板会问的灵魂问题
"怎么知道 AI 选对工具了？"

标准答案：搭评估集。准备 100 个真实用户问题，人工标注每个问题应该调哪个工具。跑系统看准确率。

准确率 ≥ 90% 才可以上线。低于 80% 回去优化 description。

03
PART · THREE
怎么处理调用结果
为啥这步最关键
API 调用 不是 100% 成功

真实世界里——网络超时、API 限流、参数错误、外部服务挂了——这些事天天发生。

能不能优雅处理失败，决定了你的 Agent 产品上线还是回炉。

1
参数校验 Validation
AI 生成的参数必须先校验再执行。城市名是不是合法、日期格式对不对、金额有没有越界。
2
失败重试 Retry
网络抖动用指数退避重试（1秒 → 2秒 → 4秒，最多 3 次）。但要识别"能重试"和"不能重试"的错误。
3
兜底回答 Fallback
真的调不通时，不要硬编结果。直接告诉用户"系统暂时无法查询，建议稍后重试或联系客服"。
4
敏感操作护栏 Guardrail
下单、转账、删数据这类不可逆操作——必须加二次确认。AI 准备调用前，先问用户"确认执行吗"。
日志必须记录
每次调用都要 log
用户原话 + AI 选的工具 + 生成的参数
API 返回结果 + 耗时 + 是否成功
最终给用户的回答

这些日志是事后分析和迭代的唯一依据。没日志，出问题就是黑盒。

面试金句 · 直接抄
被问"Function Calling 怎么设计"——
第 1 句：用 JSON Schema 定义工具，name + description + parameters
第 2 句：description 必须包含 动作 + 场景 + 边界
第 3 句：工具数量 ≤ 10 个，避免职责重叠
第 4 句：错误处理 4 策略：校验 / 重试 / 兜底 / 护栏

4 句答完，这道二面题就稳了

核心结论 · TAKEAWAY
Function Calling 不是写代码
是写好工具描述
