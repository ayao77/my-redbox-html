01
Tool Schema 到底是什么
一句话定义
Tool Schema · 告诉模型
"这个工具干嘛 / 何时用 / 参数怎么填 / 返回什么"
的结构化描述
类比 · 招聘 JD

Tool Schema 就是给模型的"招聘 JD"。

JD 写"会做事" — 来面试的五花八门;JD 写"擅长 Excel 数据透视 / 熟悉跨境财务" — 精准匹配。模型每次决定调用哪个工具,本质上就是在"面试"哪个工具合适。Schema 写得抽象模型乱选,Schema 写得具体精准命中。

三个误区 · 必须澄清

误区 1
Tool Schema 是工程师的活,PM 写需求就行。
最大的误区。描述部分是自然语言,这是产品需求,不是技术活。
误区 2
描述越短越好,省 Token。
错。Schema 走 Prompt 缓存,不变部分只算一次成本。200 字换 +20pt 准确率,完全值。
误区 3
只要 Schema 写好,模型就完美调用。
也不对。Schema 是基础,还要配并行调用、错误反馈、Eval 监控。是组合拳。
正解
Tool Schema = PM 的产品需求文档,不是工程师的代码注释。
02
Tool Schema 四要素 · 工业级标准

任何能上线的 Tool Schema 必须含 4 样 — name / description / parameters / examples。每一项都有质量标准。

01
name名字
格式 — 动词 + 对象,清晰唯一。多 Tool 时模型选不准 90% 是名字写差了。
反例get_info — 拿什么 info?
正例get_order_status — 看名字就明白
02
description描述 · PM 主战场
工业级 200-500 字。这是 PM 主要发力的地方。8 成 PM 写 20-50 字,这是首要翻车原因。
description 必含 5 件事
这个工具干什么
什么时候应该调用
什么时候不应该调用
参数简要说明
返回什么 / 怎么解读
反例(6 字)"获取订单信息"
正例(130 字)"当用户询问订单当前状态时调用。order_id 必须 16 位数字。返回物流状态、预计到达、当前位置。不要在用户没提供订单号时盲调,优先询问。只查询近 3 个月,更早调 query_archive_order。"
03
parameters参数
用 JSON Schema 精确定义。每参数 4 样必齐:类型 / 约束 / 必填可选 / 示例值。
反例order_id: string — 模型可能传订单名/备注/姓名
正例order_id: string, 16 位数字, 必填, e.g. "1234567890123456"
04
examples示例
每 Tool 配 3-5 个 few-shot。3 类示例必齐 — 一个应该调用的、一个不该调用的、一个相似 Tool 怎么区分。
必备 3 类
① 应该调 · 用户:"我订单 X 到哪了" → 调 get_order_status
② 不应该调 · 用户:"帮我查最近这单" → 先问订单号(不调)
③ 区分相似 · 用户:"我退货那单进展" → 调 search_order_by_status,不调 get_order_status

心法 — 四要素齐全,且每一项都精雕细琢。这是工业级 Schema 的底线。

03
3 个真实案例 · 不同场景
CASE A美妆电商客服 · 5 Tool 改造效率 + 准确率
改造前5 个 Tool 各 20-30 字描述 · 准确率 65% · 频繁选错工具
方案每 Tool 描述扩到 200+ 字 + 5 个 few-shot + JSON Schema 精确约束 + 开启并行调用
数据准确率 92% · 延迟 8s→3s · 月成本 反降 20%(选错重试少了)
教训Tool Schema 不是注释,是产品需求文档
CASE BSaaS 工单 Agent · 生产事故危险等级缺失
背景7 个 Tool,工程师套用 OpenAPI 注释生成 Schema
翻车模型把"重启服务"调成"停止服务" — 生产环境直接事故
复盘OpenAPI 里"重启=stop+start"没说清,"危险等级"也没标
教训Schema 必须含"使用边界"和"危险等级"。涉及生产/金额/不可逆操作的 Tool 必须明确
CASE C跨境选品 Agent · 并行调用延迟优化
背景10 个 Tool 串行调用 · 响应延迟 25 秒 · 用户等不到
方案Schema 里标注哪些 Tool 之间无依赖、可并行
数据延迟降到 8 秒 · 用户留存 +18%
教训Schema 不只描述单 Tool · 还要描述工具关系(哪些可并行/有依赖/互斥)
04
3 种典型翻车与 PM 兜底
翻车
01
描述太抽象
"获取订单"、"查询信息"、"操作系统" — 模型不知道什么时候用、边界在哪。PM 写得最多。
每个描述强制写 5 件事 — 干什么/什么时候用/什么时候不用/参数说明/返回什么。任何一项漏写不允许提交。
翻车
02
参数定义模糊
只写 type:string · 没格式约束 · 没示例值 · 没必填可选 — 模型乱传,工程师代码崩。
每个参数 4 样必齐 — 类型 / 约束 / 必填可选标记 / 示例值。如果有枚举值,枚举列表必须列全。
翻车
03
示例缺失或不典型
0 示例 · 或都是"用户问订单号就调用"这种 toy case — 模型遇复杂场景就乱。
每个 Tool 至少 3 个示例,必含 — 一个应该调用的 · 一个不应该调用的(让模型学边界) · 一个相似 Tool 怎么区分。
05
面试金句 · 主题 + 5 变种
INTERVIEW GOLD · 高频题
你怎么设计 Tool Schema?
第 1
4 要素必齐 — name 动词+对象 · description 200-500 字含 5 件事 · parameters JSON Schema 精确定义 · examples 3-5 个 few-shot。
第 2
Tool Schema 是 PM 的活不是工程师的 — 描述和示例是自然语言,是产品需求一部分。
第 3
做 AB 测试 — 同测试集跑两版,看调用准确率、看延迟、看成本。提升 20-30pt 常见。
第 4
做 监控 — 调用准确率仪表盘 / Bad Case 反馈 / 月度 Schema 复盘。是 Eval 体系一部分。
4 句答完,面试官知道你不是把 Tool Schema 当工程附属物的水货 PM。
变种 01
Tool 描述写多少字合适?
工业级 200-500 字。8 成 PM 写 20-50 字,这是首要翻车原因。OpenAI 和 Anthropic 官方推荐都是 200 字以上。
变种 02
Tool Schema 是 PM 的工作还是工程师的?
PM 主导。描述和示例是自然语言,是产品需求文档。参数定义跟工程师协作。工程师写出来的 Schema 准确率永远比 PM 写的低。
变种 03
并行调用怎么开?
Schema 标注哪些 Tool 无依赖、可并行。模型自动识别批量调用。延迟降 50-70%。GPT-4 / Claude / Gemini 都支持。
变种 04
怎么测 Tool Schema 质量?
三层 — 50-100 条调用测试集看准确率 · AB 测试新旧 Schema · Bad Case 反馈机制。是 Eval 体系一部分。
变种 05
涉及生产环境的危险 Tool 怎么处理?
Schema 描述里明确 "危险等级"和"使用边界";代码层做二次确认 wrapper;模型调用前必须用户确认。这是合规底线。
ONE-LINE
Schema 改一次 准确率拉高 +27pt
模型没变 Tool 没变

变的是 PM 把它当不当 PRD 写
