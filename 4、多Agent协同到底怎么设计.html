到底怎么设计

By NOVA · 面试最容易翻车的 AI 架构题
01
PART · ONE
主 Agent 和子 Agent 怎么分工
❌ 先讲为什么需要多 Agent
单 Agent 有 3 个死穴
死穴 1专业度上限低（一个人管法务 + 财务 + 客服，都做不精）
死穴 2上下文容易爆（所有 prompt 塞一起，烧 token）
死穴 3错误难定位（黑盒里出错，不知道是哪个环节）
✅ 解法：把全能选手拆成专科团队
行业标准方案 = Supervisor 模式

主 Agent 负责调度，子 Agent 负责干活。

02
PART · TWO
消息怎么传
💡 核心问题
多个 Agent 之间，怎么传递信息？

3 种主流模式，各有适用场景。

01
共享内存Shared Memory
最简单
原理 所有 Agent 读写同一个工作空间，像一块共享黑板。
好处 实现快，没有通信开销
坏处 容易乱——大家同时改一个东西，数据冲突
用在 小规模 demo · Agent 数量 ≤ 3
02
消息总线Message Bus
最主流 ⭐
原理 Agent 之间发结构化消息，主 Agent 转发，像微信群有群主在管。
好处 清晰可追溯 · 每条消息可日志化
坏处 需要定义消息协议（JSON Schema）
用在 生产环境首选 · LangGraph / CrewAI 都用这个
03
状态机State Machine
最稳定
原理 固定流程 A → B → C，按顺序执行，像工厂流水线。
好处 极其稳定 · 出错可定位到具体节点
坏处 不灵活——不能处理复杂分支
用在 合规要求高的场景 · 金融、医疗
💡 选型口诀

小 demo → 共享内存
生产环境 → 消息总线（默认选这个）
合规高压 → 状态机

03
PART · THREE
冲突怎么解
💡 真实场景
子 Agent 给出矛盾的答案怎么办？

用户问"这笔合同能签吗"——
法务 Agent 说"能签"，财务 Agent 说"不能签"。
谁说了算？

01
主 Agent 仲裁Supervisor Arbitration
最主流 ⭐
原理 子 Agent 意见冲突 → 主 Agent 看综合情况拍板，像老板做最终决策。
好处 简单 · 责任清晰
坏处 主 Agent 也可能判错
用在 大部分业务场景
02
投票机制Voting
适合多视角
原理 多个子 Agent 投票，少数服从多数。比如 3 个 Agent 评估代码，2 个说 OK 就过。
好处 减少单点偏见
坏处 慢 · 烧 token · 不适合实时
用在 代码评审 · 内容审核
03
优先级规则Priority Rules
合规必备
原理 预设规则——比如"安全 Agent 否决一切"。法务说不能签 → 直接停，不用商量。
好处 可控 · 可解释
坏处 需要预先设计规则
用在 金融、医疗、法律 · 合规优先场景
📏 面试金句 · 直接抄
被问"多 Agent 冲突怎么处理"——
默认上 Supervisor 仲裁
多视角评估上 投票机制
合规场景上 优先级规则
关键是 "分场景选模式" 这句话

能答出 3 种模式 + 适用场景 → 面试官立刻刮目相看

核心结论 · TAKEAWAY
多 Agent 不是堆数量
是分工 + 通信 + 仲裁三件事
