AI Agent 上线 3 天
7 个真实翻车现场

朋友的 AI 销售助手,上线第 3 天给 27 个客户开错了发票。不是 Demo 翻车,是真把事干错了。我帮他过了一遍调用日志,7 种死法,一起拆给你看。

SLACK 朋友 · CTO WED · 16:47 #direct-message-to-nova
他们的产品是 B 端 AI 销售助手,给销售总监用。 功能是用自然语言让 AI 帮你查订单、查余额、下单、开发票、退款。 听着挺常规一个 AI 项目,出问题出在 —— 那一类我们叫 Agent 的产品身上。

Agent ≠ 代理人

在 AI 圈里,Agent 特指那种你跟它说一句话,它自己决定调用哪几个工具、按什么顺序调用、把结果组合给你的 AI 助手。

只生成文字回答。
它告诉你"你应该这样做"。
真的去执行操作 —— 真的查余额、真的开发票、真的退款。
它直接干。

所以 Agent 一旦炸 —— 炸的不是"AI 回答错了",是"AI 真的把事干错了"。

📝 后面遇到 工具 就当它是"AI 能调用的一个功能",比如查余额是一个工具、开发票是一个工具。

7 CRASHES真实日志拆出来的 7 种死法

每个翻车现场 · 我用 具体场景画面 + 为什么会发生 + 怎么防 三块讲 · 上线前对一遍。

01
01
工具调用顺序错ORDER · 反着调
顺序反
🗣 销售
「帮我给客户 A 下一单 1 万元的货」
正常
先查余额 → 够了再下单
Agent
反过来 · 先调 下单 · 后调 查余额
后果
客户余额只有 8000 · 订单已生成 · 付款失败 · 客户那边看到 "订单已生成但付款失败" 一脸懵
Agent 决定调用顺序的依据是 · 你给每个工具写的 "说明书" (description) · 如果 下单 工具说明书写得有吸引力 · Agent 会优先调它。
在工具说明书里加 "前置依赖" 这一项

下单 说明书要写明:"调用前必须先调用 查余额,且余额 ≥ 订单金额"。Agent 会按这个规则自己排顺序。

02
02
Agent 自己编参数HALLUCINATE · 27 张发票主因
参数幻觉
正常
先调 查客户 · 拿到 customer_id(客户编号)→ 传给 开发票
Agent
前面没调 查客户 · 或调了没记住 · 开发票时 自己编了一个 customer_id
后果
→ 27 张发票开到了隔壁公司账上
这种 AI 没依据就自己生成内容 的行为 · 在 AI 圈叫 "幻觉" (Hallucinate)。文字幻觉无伤大雅 · Agent 幻觉一个参数 · 会把发票开到别人账上。
写入类工具加 "参数白名单校验"

开发票 / 退款 / 转账 这种写入类工具 · 参数只接受 系统认得的真实 ID。Agent 编的乱码立即报错 · 不让走通。

03
03
两个相似工具 · Agent 乱选AMBIGUOUS · 名字像就乱挑
说明书歧义
工具 A
SearchOrder 说明书写 "查找订单"(实际是查未付款)
工具 B
QueryOrder 说明书写 "查询订单"(实际是查已付款)
后果
销售来回切错单 · 自己都搞不清查的是哪一笔 · 区别全靠词义体感 · 模型自己猜
模型在工具选择上 · 看到两个相近的工具会按 "哪个名字看上去更像" 决策 · 不会按业务逻辑。
说明书写明 "我能做什么 / 我不能做什么"

SearchOrder 改成 "查未付款订单 · 仅返回 status = unpaid" / QueryOrder 改成 "查已付款历史订单 · 仅返回 status = paid"。

04
04
多步骤之间上下文丢失CONTEXT LOSS · Agent 忘了客户 B
忘了前文
🗣 销售
「老板让我给客户 B 下单」
第 1 步
Agent 调 查客户 · 拿到客户 B 信息
第 2 步
Agent 调 下单 · 把客户 B 信息丢了 · 直接拿销售自己账号下单
后果
单子下到了销售自己头上 · 跟客户 B 没关系
Agent 每一步之间只看 "上下文"(对话历史)+ "工具返回结果"。如果没把客户 B 的关键信息从结果里抽出来 明确放进上下文 · 下一步就忘了。
多步骤工具流加 "结果摘要" 步骤

Agent 调完第一步后 · 先用一句话总结关键信息 · 再开始第二步。

05
05
单点失败没兜底NO RETRY · 失败当成功
假装成功
🗣 销售
「帮客户 C 退一笔款」
第 1 次
Agent 调 退款 API · 失败(支付网关抖动)
Agent
直接告诉销售 "退款成功"
后果
销售告诉客户 C 已退款 · 客户等一周钱没到账 · 群里炸
Agent 默认认为 "工具调完了 = 成功" · 工具失败的返回值没被区分对待。
写入类调用后加 "结果确认"

支付 / 退款 / 转账 后必须再调一次 查交易状态 · 确认 status = success 才回报。fail 状态走 "重试 or 转人工"。

06
06
触发条件错位OVER-TRIGGER · 看 ≠ 取消
自作主张
🗣 销售
「帮我看看客户 D 上次那个未付款的订单」
Agent
调 查订单 看到 30 天未付款 · 自作主张 调 取消订单
后果
客户 D 的订单被取消了 · 销售只是问了一下 · 没说要取消
Agent 看到 "30 天未付款" 内部策略判断 "这是个该取消的订单" · 把 "看" 当 "取消" 来执行。
任何不可逆操作前必须 "二次确认"

Agent 不要直接调 取消订单 · 要先生成一句话:"我打算取消客户 D 的订单 #12345 · 你确认吗?" 等销售明确说 "是" 才执行。

07
工具竞争 · 总走贵的那个BIAS · 70% 走贵通道
每天损失千元
通道 A
PayChannelA · 0.6% 手续费 · 说明书写 "支付通道 · 工商银行优先"
通道 B
PayChannelB · 1.2% 手续费 · 说明书写 "支付通道 · 稳定 · 支持所有银行 · 实时到账"
Agent
70% 走了 B(说明书看起来更全能) · 每天多损失几千块
Agent 看到两个可替代工具 · 会挑 "看起来更全能" 的那个 · 不知道你的业务偏好。
说明书写明 "优先级"

PayChannelA 改成 "默认支付通道 · 优先使用 · 手续费最低 · 99% 场景适用" / PayChannelB 改成 "备用支付通道 · 仅在 A 不可用时调用"。
