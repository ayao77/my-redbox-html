Prompt vs 上下文工程
旧 Prompt 工程
调的是「怎么问」 —— 措辞、角色、示例、思维链。一次性的、对单轮提问有效。
新 上下文工程
设计的是「模型生成时手里有什么信息」 —— 记忆、检索、工具、历史怎么进窗口、怎么取舍。是贯穿整段交互的架构活。

怎么用 · 走一遍
一个真实客服 Agent,聊到第 20 轮开始答非所问 —— 看四把刀怎么救。
症状:客服 Agent 聊到第 20 轮,把别的工单信息串进来、忘了用户最初要退款、还在引用早期一句错地址。
诊断 = 分心 + 混淆 + 中毒 三连。
写
把已确认的工单事实(订单号、退款诉求)写进外部记忆,不再让它躺在对话里滚。
选
工具/知识不全量塞,按当前意图 JIT 取 —— 这一轮要退款政策,才拉退款文档。
压
每 5 轮把前文压成一段结论摘要,只留最近 3 轮原文 + 摘要。
隔
把「查物流」「查退款」拆成两个子 Agent,各自干净窗口,主 Agent 只汇总。

上下文预算 · 拍板默认值

给一套能直接抄去跟工程提的数
① token 预算上限 —— 逼着取舍,不许无限塞。默认:每个 Agent ≤ 8k;超 70%(≈5.6k)触发压缩
② 压缩策略 —— 旧历史摘要成 1 段。默认:保留最近 3 轮原文 + 1 段状态摘要,其余压掉
③ 工具/检索结果 —— 用完即清。默认:tool 结果读完即清出窗口;文档按需 JIT,不一次性灌
④ 子 Agent 上限 —— 别拆太碎。默认:≤ 3 个,每个职责单一、窗口独立
⌨ 压缩 prompt(照抄改改就能用)把以下对话历史压成一段状态摘要,只保留: ① 已确认的用户事实/偏好 ② 当前任务目标与已完成步骤 ③ 待办/未决问题。 删掉:寒暄、重复、已失效信息、试错过程。 输出 ≤ 200 字,供下一轮当上下文。
什么时候最该做
重点做
· 多轮长对话、Agent、工作流(越长越要管)
· 接了一堆工具/知识库,模型老选错
· 答案时好时坏、越聊越飘
可先放
· 一次性单轮问答,上下文很短
· 没有多工具、没有长历史
· 先把 Prompt 调好就够用的简单场景
