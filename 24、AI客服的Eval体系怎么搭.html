01
背景 · 7 成团队怎么建错的?

我接触过的 AI 客服团队,差不多 10 个里有 7 个连测试集都建错了。他们的"评估"长这样 —

3 种典型错法 — 靠手感"试了几个 case 看起来还行" · 靠用户投诉"没骂我们就是好" · 靠 Demo 跑通率"演示给老板看通了"。这都不是 Eval,是赌博。

真实数字 — 某美妆电商客服,挑了 20 条理想问题,线下准确率 90%。上线第二周,真实用户问题进来,准确率直接掉到 60%。中间发生了什么?测试集里都是"理想问题",真实用户的"奇怪问法"压根没覆盖。

那为什么 8 成团队会踩坑?3 个原因 — ① Eval 跟传统软件测试不一样(主观题不是选择题)② Eval 不是一次性活,是持续监控 ③ 不同架构 Eval 指标完全不同。任何一件没搞清楚,Eval 就建错了。

02
Eval 是什么?不是 QA,是产品工程
一句话定义
Eval · 用一组测试用例,
系统性地评估 AI 产品
输出质量的方法
类比 · 体温计 vs 出生证

Eval 是 AI 产品的体温计,不是出生证。

出生证一次性签发,签完不再变;体温计要每天测,发烧才能及时发现。如果你只在上线前做一次 Eval — 等于体检一次就再也不复诊。模型表现会漂移、用户行为会变化,你看不见。

三个误区 · 必须澄清

误区 1
做 Eval 就是准备一份测试集看准确率。错。Eval 有三层金字塔,每层用不同方法。
误区 2
LLM 当评委就够了。也不对。LLM Judge 是中间层,跟人工一致性约 85%,关键场景必须人工校准。
误区 3
Eval 只在上线前做。最致命。Eval 是上线后的持续监控,不是一次性 QA。
正解
Eval = 三层金字塔 + 持续监控 + 架构敏感的指标体系。三件事齐全才算合格。
03
Eval 三层金字塔 · 工业级标准

从下往上 — 精度递增、成本递增、覆盖比例递减。任何一层都不能省。

L1

自动断言 · Programmatic

代码自动判断 — 格式/JSON 结构/关键词/长度/敏感词。最便宜最快。适合规则明确场景。

100%覆盖
L2

LLM as Judge · 评委

GPT-4 当评委 — 准确性/相关性/有用性/安全性多维打分。适合主观性强、规则难写的场景。

10-20%覆盖
L3

人工抽检 · Gold

人类专家作金标准。校准前两层、迭代 Judge Prompt。最贵最准。

1-5%覆盖

三层比例你要记住 — 100% / 10-20% / 1-5%。越往上越窄,精度越往上越高。缺一档质量都管不住。

04

上一期讲 RAG / Agent / 混合架构选型。这三种架构对应的 Eval 指标完全不同,盲目套用单一指标会判断错。

05
案例 · 美妆客服 4 阶段 进化

我对接的真实项目 — 某美妆电商客服,从零到工业级 Eval 体系,完整 4 阶段 进化路径。

阶段 1
裸跑
20 条手挑测试集

上线前看着还行,实际第 2 周客诉量翻 3 倍,30% 真实问题在测试集里完全没覆盖。
教训 · 靠手感选测试集 = 自欺欺人

客诉翻 3 倍
阶段 2
+自动断言
500 条真实流量 + L1

覆盖 95% 真实意图分布。加格式校验、敏感词、长度、合规表述检查。
局限 · "语气不友好"、"答案不够详尽"等主观问题抓不到

客诉降 40%
阶段 3
+LLM Judge
GPT-4 4 维度打分 + L2

从准确性、友好度、详尽度、安全性 4 维打分。每次发版上线前 500 条 Eval 全跑,低于阈值不发版。

回归 bug降 80%
阶段 4
+人工抽检
每周 200 条人工 + L3

客服主管打分,对比 LLM Judge,差异大的 case 进 Eval 集回看,迭代 Judge Prompt。

一致性88%

PM 一句话心法 — Eval 数据集是产品资产,Eval 体系是产品工程能力。整个过程 3-6 个月,这是真实搭建路径。

06
3 种典型翻车与 PM 兜底
翻车
01
测试集选偏 · Biased Dataset
都是"理想问题",真实用户"垃圾问题"覆盖不到。线下 90 → 上线 60。
  1. 真实流量抽样,按意图分布、复杂度分布覆盖
  2. 每月新增 hard case
  3. Bad Case 反馈机制必须打通(用户点踩 → Eval 集)
翻车
02
LLM Judge 自己跑偏 · Judge Drift
Judge Prompt 写得差 + 用便宜模型当评委 → 评分完全失真。
  1. Judge Prompt 像 Tool Schema 一样精雕细琢
  2. 评委用最好的模型(GPT-4/Claude 系列),别省钱
  3. 每周抽样校准跟人工的一致性
翻车
03
只上线前跑 Eval · No Continuous Monitor
上线前通过,上线后模型表现漂移你不知道。等客诉飙升才发现,损失已造成。
  1. Eval 每天跑 — 核心指标趋势监控
  2. 准确率掉 2pt 自动告警
  3. 每月做一次全维度"Eval 体检"
07
面试金句 · 主题 + 5 变种
INTERVIEW GOLD · 高频题
你怎么做 AI 产品的 Eval?
第 1
用三层金字塔 — 自动断言 100% / LLM Judge 10-20% / 人工抽检 1-5%。
第 2
测试集真实流量抽样,按意图分布覆盖。Bad Case 反馈必须打通。
第 3
架构不同 Eval 不同 — Prompt 看输出/RAG 分两层/Agent 看四维。
第 4
Eval 是持续监控,不是一次性 QA。每天跑/每周看/每月校准。
4 句答完,面试官知道你不是把 Eval 当 QA 看的水货 PM,是真把 AI 当工程系统管的人。
变种 01
RAG 和 Agent 的 Eval 区别是什么?
RAG 分开看检索 + 生成两层(召回准确率 + 幻觉率);Agent 看四维(完成率/工具/轮数/成本)。本质区别 — RAG 静态问答 vs Agent 动态过程。
变种 02
测试集怎么建?
从真实流量按意图分布抽样;Hard Case 让业务方挑;Bad Case 反馈机制必须打通;每月新增 hard case。
变种 03
LLM 当评委靠谱吗?
GPT-4 跟人工一致性约 85%。要满足三个条件 — Judge Prompt 精雕细琢、用最好的模型、每周抽样校准一致性。
变种 04
Eval 多久跑一次?
不是一次性。自动断言每次发版 100% 跑;LLM Judge 每天跑;人工抽检每周跑;全维度体检每月跑。
变种 05
Eval 跟传统 QA 有什么区别?
传统 QA 选择题对错二分类;AI Eval 主观题分维度打分,且要持续更新测试集、按场景分布抽样、做漂移监控。完全不一样。
