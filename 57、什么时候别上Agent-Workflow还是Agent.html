2026 最贵的错

产品经理最容易犯、也最贵的错，就是什么都想上 Agent。你那个 Agent 干的活，很多其实一条写死的工作流 + 一次大模型调用就够了——便宜大概 100 倍、快几十倍、还稳得多。

✓Workflow

步骤写死在代码里，你掌控每一步。可预测、可追溯、成本方差近零。

!Agent

大模型自己决定下一步调什么工具。灵活，但不可预测、贵、难 debug。

一句判据 · 记死它 这事的步骤，能不能提前用代码写死？能写死 → 用 Workflow；写不死、步数没法预测 → 才用 Agent。 Anthropic 官方框架也是这句：从最简单的开始，Agent 是最后手段，不是默认选项。
为什么这么多人栽 跟「demo 死亡之谷」一个病根：demo 惊艳 + Agent 框架火，团队头脑一热，把本该是工作流的东西全改成自治 Agent → 上线就是又贵、又慢、又不稳、还难 debug。
4 道判断题
题 01

能不能写死：流程图每个分支画得出来吗

画得出全部分支 → 就该是工作流，别上 Agent。Agent 留给那种你根本画不出全部分支、步数不可预测的开放问题。
需求一来就想上 Agent，觉得酷。
先画流程图，能画死就写死——一次调用 + 几个固定步骤搞定。
把流程画成流程图，每个分支都画得出来，就用工作流。
题 02

一两次调用够不够

1-2 次大模型调用能完成的任务，上 Agent 多花 3.87 倍成本、精度提升为 0。别为「自治」的名头白烧三倍多的钱。
包个 Agent 显得高级。
先拿一次调用 + 几个工具试，能过验收就一次解决，绝不多此一举。
先用最土的一次调用试，能过就别包 Agent。
题 03

成本和延迟兜得住吗

工作流：单次约 0.001-0.02 美元、800ms；Agent：迭代 5-10 轮约 0.2-2 美元、10-30 秒。贵 10-100 倍、慢几十倍，成本还随输入波动 5-10 倍。真实事故：没设熔断，Agent 一夜循环烧掉 437 美元。
不算账，直接上。
算清单次成本 × 用量，并给 Agent 设最大轮数 + 成本熔断。
上 Agent 必须设最大轮数 + 成本熔断，不然一个死循环够你喝一壶。
题 04

多步可靠性 + 有没有人运维

哪怕每步 99% 可靠，10 步串起来只剩 90%，链越长掉越快。Agent 还会幻觉、在失败动作上死循环、撑爆上下文、用错工具——得有人盯、有监控。
能自治就全自治，省心。
能减一步就减一步；上线前先定谁盯它、崩了谁修。
Agent 每多一步，就多一分翻车概率，能减步就减步。
重构招

把又贵又不稳的 Agent 拉回工作流

统计它的决策，多半 90% 落在 4-5 种固定模式上，真正要它灵活发挥的只有剩下 10%。把这 90% 写成工作流分支、用代码定死，只在剩下 10% 才调一次大模型 → API 成本降 60-80%，还把可预测性拿回来。

Anthropic 5 种工作流模式

绝大多数你以为要上 Agent 的场景，这 5 种就覆盖了：

1
提示链：一步接一步，前一步的输出喂给下一步。
2
路由：先判断类型分流，再交给对应的处理。
3
并行：几路同时跑，最后把结果合并。
4
编排-工人：一个总控把活拆开，派给几个执行。
5
评估-优化：生成完自己打个分，不够好再改一版。
那什么时候才该上 Agent

三个特征同时满足，才是 Agent 的主场

开放式（写不死流程）+ 步数不可预测 + 但你能验证进度。比如客服 Agent 带能真正操作的工具、代码 Agent 带测试、computer-use 带截图当标准答案。少一个，都先用工作流。

2026 最值钱的能力，不是什么都敢上 Agent，是知道什么时候别上
▸默认从最简单开始　一次调用能过就一次调用，Agent 是最后手段
▸先问能不能写死　能写死就绝不上 Agent
▸敢拦 + 三笔账　拿成本、延迟、可靠性说清为什么不该上
Agent 不是越自治越高级，选对了才高级。
