01为什么需要:基础 RAG 三个不够
老 RAG 查一次就答,问题就出在这"一次"。
只查一次
这次没检索到对的资料,也照样硬塞硬编,翻车。
不会改写
用户问得模糊/没搜到,它不会换个问法再搜。
答不了多跳
"对比 A 和 B 的退款政策"要跨多段多库,单次拿不全。
02是什么:从"查一次"到一个闭环
基础 RAG = 照着翻到的那一页答;Agentic RAG = 会自己查资料的研究员。

034 个新能力(基础 RAG 没有)
＋ 举例
检索决策
Agent 先判断要不要查、查哪个库。简单问题不查,直接答,省钱省延迟。
像"今天几号""你是谁"这种,Agent 判断根本不用查知识库,直接答——省下一次检索、省钱省延迟。
＋ 举例
查询改写
第一次没搜到好结果,自动改写成更好的搜索词再搜。例:"这个能退吗"→"X 产品七天无理由退货政策"。
用户原话"它咋退啊"指代不清、搜不到 → 改写成"X 产品 七天无理由退货 流程",再查一次就命中。
＋ 举例
相关性评估
给检索回来的每条打分,低于阈值丢掉;好的太少就触发"换问法再来一轮"。
搜回 10 条,7 条是别的产品 → 全丢;剩 3 条还不够,就触发"换个问法再来一轮"。
＋ 举例
多跳综合
跨多段、多库的复杂问题,分步检索、逐步把答案拼起来。
"A 和 B 哪个退款更快" → 先查 A 的时效、再查 B 的时效,最后把两段拼成对比结论。
5 个落地模式(面试加分)
迭代检索查询分解假设驱动跨库三角验证证据加权综合
04怎么用:一个真实问题走一遍
问题:"对比 A 和 B 两款产品的退款政策有什么不同"(多跳)。
1
Agent 先把复杂问题拆成两个子问题:A 的退款政策、B 的退款政策。查询分解
2
分别去查。A 搜到了;B 第一次没搜全,自动把"B 退款"改写成"B 七天无理由退货条款"再搜一次,搜到了。查询改写
3
给搜回来的每条打分,扔掉不相关的,确认 A、B 政策都齐了。相关性评估
4
够了,把两边拼起来生成对比答案。多跳综合
技术落在哪 · 用什么搭
它落在你原来 RAG 的 "检索"和"生成"中间,多包一层 Agent 编排。
编排:LangGraph 或 LlamaIndex 的 agent 模块。
底层不变:还是你那套 向量库 + reranker。
拆解 / 改写 / 打分这些判断:交给一个大模型来做。
不用推翻原来的 RAG——是在它外面套一个"会思考"的壳。
PM 该定的 3 件事:① 这个问题要不要触发多轮 ② 最多查几轮(防无限查烧钱) ③ 什么时候算"够了"该停。这三件定不好,要么答不全,要么烧钱还慢。

被问"你那套 RAG 不够用怎么办",按 4 句答:
1先看是不是多跳——简单单跳,基础 RAG 就够,别上 Agentic。
2再加 4 个能力——检索决策、查询改写、相关性评估、多跳综合。
3必设护栏——最大检索轮数、成本上限,防它无限查。
4按场景选——Agentic 贵 3-10 倍,只在高价值多跳场景上。
4 句答完,面试官知道你不是只会喊新名词,是懂什么时候该花这个钱的人。
